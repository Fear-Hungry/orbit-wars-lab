# Experimentos

## Sobre a competição (Orbit Wars)

Competição Kaggle de simulação (`orbit-wars`): um jogo de estratégia orbital em
tempo real, jogado em **2 ou 4 jogadores**. Cada jogador controla planetas que
**orbitam e produzem naves**; a ação é lançar frotas — `[from_planet_id,
direction_angle, num_ships]` — para **capturar, reforçar ou atacar** outros
planetas. O motor resolve movimento contínuo, colisão (borda/sol/planetas),
rotação planetária, combate (maior força vence) e **cometas** que aparecem
periodicamente. O objetivo é dominar território/produção até o fim do episódio.

Constantes do ambiente oficial (`kaggle-environments`, fonte de verdade em
[`docs/PARITY.md`](docs/PARITY.md)): `episodeSteps=500`, `actTimeout=1s`,
`agents=[2,4]`, `shipSpeed=6.0`, `cometSpeed=4.0`.

Regras do jogo da competição que moldam todo experimento aqui:

- **Submissão = `agent(obs)` Python puro, rodando em CPU** com `actTimeout=1s` por
  decisão. Nada da submissão pode depender do motor Rust (invariante D10/D11). Treino
  pode usar GPU; inferência/submissão não.
- **Pontuação** é um leaderboard estilo Elo que **infla no começo e só estabiliza
  após ~1h** — nunca julgar uma submissão pelo score imediato.
- **Meta = TOP 5.** O agente público **Producer (~1200 LB)** é o **piso (entrada),
  não o teto**: bater o Producer é necessário, não suficiente. Régua decisora local
  = margem normalizada vs Producer a 96 seeds (ver "Baseline atual" abaixo).

Visão geral do projeto em [`README.md`](README.md); inteligência competitiva em
[`docs/COMPETITIVE_INTEL.md`](docs/COMPETITIVE_INTEL.md).

Registro curto para comparar ideias de heurística, decoder, modelo e liga.

## Como registrar

Toda hipótese testada entra como **uma linha na tabela objetiva** do "Painel objetivo"
(abaixo). Mudanças no agente só podem ser commitadas depois de registrar a margem
normalizada antes/depois contra a régua decisora: Producer nomeado/local. `submission_v_old.py`,
`greedy` e `rush` entram só como sanity de crash/legalidade.

Schema obrigatório (uma linha por experimento):

```text
| exp_id | hypothesis | cv | public_lb | delta_cv | seed | files_changed | risk | submit? |
|---|---:|---:|---:|---:|---|---|---|---|
```

Legenda das colunas (semântica **desta** liga de simulação, não Kaggle supervisionado):

| coluna | significado neste projeto |
|---|---|
| `exp_id` | id da família/tentativa (`E3`, `E4`, `C1a`, `1b-iii`…) |
| `hypothesis` | o que se testou, em uma linha |
| `cv` | **métrica decisora LOCAL** = `mean_score_margin` vs Producer (análogo do CV; **não há** cross-validation real aqui) |
| `public_lb` | score **Elo** no Kaggle — preenche **só se submetido** (maioria é local-only → vazio) |
| `delta_cv` | Δ da margem vs o baseline anterior |
| `seed` | nº de **seeds** do benchmark (96 = gate decisor; 4/16 = smoke/triagem, não promove) |
| `files_changed` | arquivos tocados |
| `risk` | risco operacional (timeout / custo de decisão / estado global por worker) |
| `submit?` | decisão: `sim` / `não` / motivo curto da rejeição |

O log detalhado em prosa (seção "Resultados recentes", o comando completo + antes/depois)
continua como **audit trail** que alimenta cada linha da tabela — não substitui a tabela.

Métricas úteis:

- `2p_win_rate` por oponente;
- `4p_win_rate`;
- `mean_score_margin`;
- `invalid_action_rate`;
- `timeout_rate`;
- `crash_rate`;
- pior seed ou lineup ruim.
